# 拼图相框功能开发设计规范（最终版）

# 一、项目背景

本项目是一个基于 **Vite + 原生 JavaScript + Canvas 2D API** 开发的拼图照片处理工具。

目前项目已经拥有一套稳定、完整、经过大量测试验证的图片裁剪系统。

该系统已经能够正确完成：

* 图片裁剪
* 图片缩放（Zoom）
* 图片移动（Offset）
* 图片旋转（Rotation）
* 白边填充（FillColor）
* 不同比例适配
* 下载导出

这些功能已经完全满足生产使用。

**此次开发仅新增一个功能：**

> **相框效果预览（Frame Preview）**

注意：

这不是重构图片处理模块。

也不是重新设计 renderImage()。

更不是重新计算图片比例。

整个图片处理逻辑保持不变。

---

# 二、本次开发目标

新增一个透明 PNG 相框预览功能。

用户打开"相框效果"以后：

已经裁剪完成的拼图图片

↓

整体缩放（Bitmap Scale）

↓

放入透明 PNG 相框中

↓

显示最终效果。

下载功能仍然保持：

**下载裁剪后的图片，不下载相框效果图。**

因此：

相框只是一个预览功能。

---

# 三、整个项目必须拆分成两个独立模块

## 模块一

Image Processing

职责：

负责图片处理。

包括：

* 图片裁剪
* 图片缩放
* 图片移动
* 图片旋转
* 白边填充

输出：

PuzzleCanvas

---

## 模块二

Frame Display

职责：

负责图片显示。

包括：

* 加载透明PNG相框
* 读取内框坐标
* 把 PuzzleCanvas 放进相框
* 绘制最终效果

它绝对不能：

参与图片处理。

---

# 四、renderImage() 是唯一图片处理入口

整个项目：

renderImage()

只能调用一次。

它负责：

* Zoom
* Offset
* Rotation
* FillColor
* Crop

最终输出：

PuzzleCanvas

例如：

```text
原图

↓

renderImage()

↓

PuzzleCanvas
```

PuzzleCanvas 是整个项目唯一的图片处理结果。

# 十二、drawImage 使用规范

整个 Frame 模块：

最多只有两次 drawImage。

第一次：

绘制：

PuzzleCanvas

例如：

```javascript
ctx.drawImage(
    PuzzleCanvas,
    0,
    0,
    PuzzleCanvas.width,
    PuzzleCanvas.height,
    innerLeft,
    innerTop,
    innerWidth,
    innerHeight
);
```

第二次：

```javascript
ctx.drawImage(
    framePNG,
    0,
    0
);
```

除此之外：

禁止再次 drawImage。

禁止二次缩放。

禁止再次裁剪。

# 十七、最终目标（修正版）

最终的相框功能必须遵循以下原则：

## 第一步（唯一一次图片处理）

调用现有的 `renderImage()`。

生成最终拼图：

```text
───────────────
│              │
│ PuzzleCanvas │
│              │
───────────────
```

此时：

* 白边已经生成
* 图片已经裁剪
* 图片已经缩放
* 图片已经移动
* 图片已经旋转
* 图片已经完成所有用户编辑

这是整个流程唯一允许调用 `renderImage()` 的地方。

---

## 第二步（唯一允许的变换）

允许把整个 `PuzzleCanvas`：

作为一个整体进行 **Bitmap Scale（位图整体缩放）**。

例如：

```text
2000×1500

↓

Bitmap Scale

↓

1728×1296
```

注意：

缩放的是：

整个 Bitmap。

不是：

重新计算图片。

因此：

不会改变：

* 图片裁剪结果
* 图片内容
* 白边比例
* 图片位置
* 图片旋转
* 用户调整结果

用户看到的内容保持完全一致，只是整体显示尺寸变小，以适配相框内框。

---

## 第三步（严禁再次图片处理）

从 PuzzleCanvas 创建完成以后：

严禁再次调用：

renderImage()

严禁再次执行：

* Zoom
* Offset
* Rotation
* Crop
* FillColor

严禁重新计算：

* imageWidth
* imageHeight
* drawWidth
* drawHeight

严禁重新适配图片比例。

---

## 第四步（最终合成）

整个相框模块最终只能执行：

```text
Bitmap Scale

↓

draw PuzzleCanvas

↓

draw PNG Frame
```

整个模块只是一个：

**Display（显示）模块。**

不是：

**Image Processing（图片处理）模块。**

---

# 十八、架构原则（最重要）

请始终牢记：

> **相框模块是一个纯 Display（显示）模块，而不是 Image Processing（图片处理）模块。**

它只能接收已经生成好的 `PuzzleCanvas`。

不能参与任何图片编辑。

不能重新计算任何图片参数。

不能重新裁剪图片。

不能重新生成图片。

它唯一允许执行的操作只有：

1. 对 `PuzzleCanvas` 做一次 **Bitmap Scale（位图整体缩放）**；
2. 将缩放后的 `PuzzleCanvas` 绘制到相框内框坐标；
3. 将透明 PNG 相框覆盖到最上层。

除此之外，不允许存在任何图片处理逻辑。

---

本次开发目标不是“修修补补”，而是以清晰、稳定、可维护的架构，实现一个**完全独立的相框显示模块**。
